SÉSAME — Récapitulatif des travaux

Application de prescription rémunérée · Backend Node/Express · Mobile React Native · Dashboard Admin

Journée du 11 juin 2026  ·  7 chantiers  ·  Document interne
Résumé exécutif — La journée a couvert deux grands axes : (1) la refonte des commissions des entreprises (Ambassadeurs Moraux) côté admin et mobile, et (2) une passe de sécurité complète rendant l'application prête au déploiement (authentification, droits d'accès, WebSocket, durcissement). Tous les éléments ci-dessous sont implémentés, compilés et vérifiés.

1. Commissions des entreprises (Ambassadeurs Moraux) FAIT

Correction de calcul

  • Le CA et la commission d'une entreprise comptaient uniquement ses courses directes et ignoraient celles de ses employés (sous-comptes) — corrigé côté admin et côté app mobile.
  • Désormais : commission = 10 % du CA de l'entreprise + tous ses employés, sans double comptage.

Nouvelles fonctionnalités de l'onglet Commissions (admin)

  • Ajout de l'onglet « Commissions » dans le menu (il existait dans le code mais n'était pas affiché).
  • Affichage du nom de l'entreprise à côté du responsable légal.
  • Sélecteur de mois + ouverture par défaut sur le mois précédent (paiement à terme échu, conforme aux specs).
  • Colonne « Statut versement » (À verser / Versé le JJ/MM) avec nouvelle table virements_commissions.
  • Bouton « Déclencher » qui enregistre réellement le versement — strict & idempotent (ne re-paie jamais une entreprise déjà payée).
  • Bandeaux pédagogiques expliquant le cycle de paiement à l'administrateur.

2. Améliorations d'affichage (mobile) FAIT

  • L'entreprise (compte Moral) peut commander des courses — vérifié de bout en bout (déjà supporté, confirmé).
  • Écran d'accueil : la section « Mon équipe » affiche désormais un aperçu limité (3 employés max + « +N autres ») au lieu de toute la liste, qui débordait sur le reste de l'écran.

3. Authentification & sessions SÉCURITÉ

  • Le token expirait après 2 h sans jamais se rafraîchir → spam de logs [token invalide]. Corrigé.
  • Token porté à 24 h + rafraîchissement automatique (intercepteur axios côté mobile, avec dédoublonnage).
  • /auth/refresh accepte un token expiré (signature valide) mais refuse au-delà de 30 jours → reconnexion forcée.

4. Sécurité des routes — authentification & droits d'accès SÉCURITÉ

Faille majeure découverte et corrigée : le serveur ne faisait que logger les tokens sans jamais bloquer. N'importe qui pouvait lire/modifier les données de n'importe quel utilisateur en changeant un identifiant dans l'URL.
  • Authentification obligatoire (requireAuth) : toute requête sans token valide reçoit 401.
  • Vérification de propriété : un utilisateur n'accède qu'à ses propres données (ambassadeur, chauffeur, courses, bons, chat) — sinon 403.
  • Routes publiques préservées (page web fournisseur, paramètres publics, authentification).

5. WebSocket du chat SÉCURITÉ + RÉPARÉ

  • Le chat temps réel n'était pas authentifié → ajout de la vérification du token + appartenance à la course à la connexion.
  • Bug historique corrigé : à cause d'un match de chemin exact dans la librairie, le WebSocket ne s'était jamais connecté (le chat tournait en secours sur du « polling »). Le temps réel fonctionne désormais.

6. Audit de sécurité & durcissement déploiement SÉCURITÉ

Audit complet réalisé (secrets, injection SQL, brute-force, upload, fuites, logique métier). Aucune injection SQL (100 % requêtes paramétrées). Corrections appliquées :

MesureDétail
Secret JWTValidé au démarrage (≥ 32 car.), refus de démarrer sinon — plus de valeur par défaut faible
Anti-brute-forceRate-limiting sur login, login admin, mot de passe oublié et validation du code 4 chiffres
Codes sécuriséscrypto.randomInt / randomBytes (reset, code de validation, token QR) au lieu de Math.random
En-têtes HTTPAjout de helmet
CORSRestreignable par variable d'environnement (CORS_ORIGINS)
Upload de fichiersFiltre de type (jpeg / png / pdf uniquement)
Fuites d'informationMessages d'erreur génériques en prod + plus de données personnelles dans les logs

7. Reste à faire / à configurer au déploiement

  • Variables d'environnement (prod) : NODE_ENV=production, JWT_SECRET, CORS_ORIGINS, ADMIN_EMAIL/ADMIN_PASSWORD (fort), secrets Stripe & Supabase.
  • SQL à exécuter sur la base : création de la table virements_commissions.
  • Ne jamais lancer npm run seed en production (mots de passe de test faibles).
  • Confort (non bloquant) : persistance du token pour rester connecté entre deux ouvertures de l'app.
  • Intégration bancaire : le bouton « Déclencher les virements » enregistre l'ordre mais le virement SEPA réel (Stripe) reste à brancher.
